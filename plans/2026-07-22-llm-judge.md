## Why / what changes for the user

Today stratus's only pre-publish opinion on an original is `POST /x/posts/rewrite`: three unranked rewrites you eyeball and guess between. There is no answer to "is this post actually any good", and every quality signal it has is post-hoc (the Playbook, n≥20, weeks late) or about other people's tweets (`replyBand`).

After this plan: a **Run judge** button under the Composer textarea makes one call and comes back with an overall 0–100 plus twelve named dimensions (replies, profile clicks, bookmark value, dwell, voice match, negative risk, answer effort, stranger answerability, **status dependency** — does this need a famous bio to land — reply-vs-quote orientation, impressions, audience match against your active niche and pillars), a one-line headline, strengths/improvements, and up to twelve **anchored fixes**: an exact quote from your draft, a severity, and a one-line recommendation. Clicking a fix selects that span in the textarea. **Apply all fixes** rewrites the draft applying every one of them, re-judges the result, and keeps it **only if it scores better** — otherwise your version comes back untouched. Editing the draft clears the verdict, so a verdict always describes the exact text it judged.

And in the same phase, the Playbook gains the cell that says whether any of it is real: own measured originals bucketed by the verdict band they were judged at, gated n≥20, stating honestly either way whether the judge's `overall` predicts views and profile clicks. x-builder — where this rubric comes from — **cannot** run that check: its verdict is never persisted. Stratus's is, keyed to the exact text, so the question is answerable in weeks.

## Design

**Provenance / licensing.** `creynir/x-builder` has **no LICENSE file** → default all rights reserved. The *rubric dimensions*, the *derived-verdict* idea, the *never-worse guard*, and the *quote-anchoring* algorithm are unprotectable ideas; their exact prompt strings, schemas, label copy and file layout are expression. **Every module here is written from the dimension list in the source study with our own prompt copy, our own schema, and our own naming**, carrying a header comment citing the study. This is also the better outcome — the rubric has to be niche-grounded for stratus anyway, and their `voiceMatch` is deliberately person-agnostic where ours is anchored on the active niche.

**Data — one new table, no columns on existing tables.**

```sql
-- migration 0018 (DDL only, no seed — the drizzle-kit dropped-seed trap can't apply)
CREATE TABLE draft_judgments (
  id                 TEXT PRIMARY KEY,          -- crypto.randomUUID()
  text_hash          TEXT NOT NULL,             -- sha256 of normalizeJudgeText(text)
  text               TEXT NOT NULL,             -- exactly what was judged (§7.18 transparency)
  surface            TEXT NOT NULL DEFAULT 'post',
  verdict            TEXT NOT NULL,             -- derived band, never model-supplied
  overall            INTEGER NOT NULL,
  scores             TEXT NOT NULL,             -- json, all 13 dimensions
  annotations        TEXT NOT NULL,             -- json [{quote,severity,recommendation}]
  strengths          TEXT, improvements TEXT,   -- json
  confidence         TEXT,
  model              TEXT NOT NULL, provider TEXT NOT NULL,
  cost_usd           REAL,
  parent_judgment_id TEXT,                      -- the re-judge of an applied rewrite points at its original
  judged_at          INTEGER NOT NULL
);
CREATE INDEX draft_judgments_text_hash_idx ON draft_judgments (text_hash);
```

**The verdict↔outcome link is a read-time text hash, not a stored foreign key** (§7.12). `normalizeJudgeText` collapses whitespace and trims (the same reading as the harvest reconcile's text-match fallback); the Playbook loader hashes every own published post and looks it up in a Map built from one `draft_judgments` SELECT. This is *semantically* right, not a shortcut: a verdict is only valid for the exact text it judged (the reset-on-edit rule), so a post edited after judging **should** read as unjudged. It also covers hand-written-and-judged posts and needs no migration on `scheduled_posts` or `posts_published`.

**Pure logic — two modules.**
- `src/shared/judge.ts` (canonical, zero-dependency, extension re-export shim per §7.27 — the `replyBand.ts`/`channelSuggest.ts` arrangement): the `JudgeScores`/`JudgeAnnotation`/`JudgeVerdict` types, `JUDGE_DIMENSIONS`, `deriveVerdictBand(overall)` (≥85 `post_now` / ≥70 `slight_rework` / ≥40 `major_rework` / else `do_not_post`), `deriveApproved`, and **`locateAnnotations(text, annotations)`** — the DOM-free port of x-builder's quote-location pass: left-to-right `indexOf` from a consumed offset, unmatched quotes **silently dropped**, returning `{from, to, quote, severity, recommendation}[]`. The panel uses the offsets with `setSelectionRange`; no rect mapping, no mirror div (stratus's composer is a plain `<textarea>` — see the study's §J1).
- `src/x/judge/prompt.ts` (server-only): `JUDGE_PROMPT_TEMPLATE`, `JUDGE_SCHEMA`, `parseVerdict(raw)`, `buildJudgeInput({draft, grounding?, template?})`, `normalizeJudgeText`/`judgeTextHash`. TS-only, **no `.md`, no byte-sync test** (AI-layer Decision 9 — that convention is reserved for post/reply/thread).

**Routes** — `src/x/routes/judge.ts`, **LLM-gated mount** beside `drafter` in `src/x/index.ts` (§7.22).

| Endpoint | Body | Success | Errors |
|---|---|---|---|
| `POST /x/judge` | `{text (1–2000), surface? ('post'), model?, provider?}` | `{id, textHash, verdict, approved, confidence, headline, scores{13}, annotations[], strengths[], improvements[], model, provider, costUsd, requestId}` | 400 `invalid_body`/`invalid_text`/`invalid_surface`/`invalid_model`/`invalid_provider`; 503 `llm_not_configured`; 502 `judge_parse_error`/`judge_invalid`/`judge_failed` |
| `POST /x/judge/apply` | `{judgmentId, text, model?, provider?}` | `{text, improved, judgmentId, verdict, approved, scores, annotations, strengths, improvements, model, costUsd}` | 400 as above + `invalid_judgment_id`; **404 `judgment_not_found`**; **409 `stale_verdict`** (hash mismatch); 503/502 as above |

`apply` is **two** calls, not x-builder's three: the caller already paid for the verdict and stratus stored it, so the chain is rewrite-from-annotations → re-judge → never-worse compare. Persisting the verdict buys a call back.

**Extension** — `extension/src/judge.ts` shim + an `extension/tsconfig.app.json` `include` entry; a judge block in `Composer.tsx` under the single-post textarea, composed from `src/sidepanel/ui/` primitives (UI.10 / D7 standing).

**Measurement (JD.7, non-optional, same phase as the tool)** — `buildJudgeEffectiveness` in `src/x/playbook.ts`: own **non-reply** originals bucketed by the verdict band their text was judged at, plus an `unjudged` bucket, each cell gated `DEFAULT_MIN_CELL_N`, and a `spread` (top band ÷ bottom band, views and profile visits) **only when both bands clear the gate** — the `ratio()` discipline every other lift uses.

## Decisions taken

1. **On demand, one draft, never automatic.** x-builder auto-judges all 3 generated candidates in parallel (`generate-ideas-service.ts:489`); it can, because its LLM is a CLI subprocess on a subscription (`process-runner.ts:1`) with $0 marginal cost. Stratus pays per token: that would turn a $0.006 drafter call into ~$0.016 for verdicts on two drafts you will discard. A manual button on the draft you are about to schedule is the whole cost argument.
2. **No judge on replies. Ever, in this plan.** Judging 3 variants × 10–20 replies/day is **$3–6/month** — a 3–5× increase of the entire LLM budget on the surface with the least headroom. x-builder's own `defaultReplyContextPolicy` (`unified-generation-context.ts:43-54`) sets `forbidJudge: true, forbidReachScoring: true`; that is independent confirmation, not just a budget excuse. The `surface` column exists so a future decision can add `'reply'` without a migration — nothing writes it in v1.
3. **The verdict is derived, never model-supplied.** The schema omits `verdict` entirely; the route computes it from `scores.overall` and spreads it last, so the label and the number can never disagree. Three lines, and it removes a whole class of inconsistency (x-builder's `judge-draft-service.ts:19`, `:148-157`).
4. **`overall` is a diagnostic, not a target.** No surface sorts, gates, auto-selects, blocks Schedule, or refuses on it. This matters more than it looks: the rubric's `answerEffort: 100 = a one-word answer` and `strangerAnswerability: 100 = anyone can reply` are, maximised, the same reply-farm gradient the static-engine study flagged in x-builder's format table. `negativeRisk` balances them only while a human is reading both. Same decision-4 as the static coach, for the same reason.
5. **`audienceMatch` and `voiceMatch` are grounded on the active niche, not a free-text setting.** x-builder anchors `audienceMatch` on a 600-char `accountProfile` the user types once, and returns **null** when it's absent; `voiceMatch` is deliberately person-agnostic ("generic authentic human voice"). Stratus stamps `loadActiveNicheSafe()`'s persona + beliefs + the active pillar labels at the **variable tail** (§7.15/§7.16, server-stamped only), so both dimensions become *your*-voice/*your*-audience checks. The `me` layer block is deliberately **not** injected — it's about specificity, not audience fit, and it would grow the prompt for no dimension that reads it.
6. **The verdict↔post link is a text hash computed at read time**, not a `judgment_id` column on `scheduled_posts` (§7.12, and the reset-on-edit semantics make it the *correct* join — see Design).
7. **Two-call apply, with the never-worse guard verbatim in spirit.** `rewriteOverall <= originalOverall` (strictly `<=`) returns the ORIGINAL text and the ORIGINAL verdict with `improved: false`; the returned verdict always describes the returned text. Any step failing is a 502 — the route can never hand back a rewrite of unknown quality (`apply-judge-suggestions-service.ts:240-247`).
8. **Reuse the existing rewrite prompt family, don't fork it.** `judge-rewrite` is a new registry key whose template carries `{{DRAFT}}`, `{{FIXES}}`, `{{IMPROVEMENTS}}` — the annotation-driven sibling of `rewrite`. `POST /x/posts/rewrite` (AI.8) is untouched: it stays the cheap no-verdict path.
9. **A failed persist never loses a paid verdict** (§7.8): the `draft_judgments` insert is wrapped like `persistRadarDrafts`; the route returns the verdict regardless, with `id: null`. The money is already spent — refusing to return it would be strictly worse.
10. **No judged-quality reach multiplier.** x-builder feeds the judged `impressions`/`replies` scores into its reach estimator through four constants all marked `// CALIBRATE` (`prediction-estimator.ts:84-89`). Converting a model's opinion into an impression forecast via invented bounds is exactly what the static-coach plan refused for L3. Not ported, not later.
11. **Provider is not forced, but always recorded.** A judge running on the same model that wrote the draft will systematically like its own output. `askLLM` already takes a per-call `provider` override at the highest precedence, so the panel exposes it and the route stamps `model`+`provider` on **every** row — the risk is documented and measurable rather than silently baked in. Forcing OpenRouter would 503 for a user with only `XAI_API_KEY` set.

## Done when

- [ ] Typing a draft in the Composer, clicking **Run judge**, and getting back an overall + band + twelve collapsed dimensions + a fix list, in one call, in under ~15 s.
- [ ] Clicking a fix row selects exactly that phrase in the textarea; a fix whose quote isn't in the draft simply doesn't render a row (no crash, no empty row).
- [ ] Editing one character clears the verdict; re-running produces a verdict whose stored `text_hash` matches the new text.
- [ ] **Apply all fixes** either replaces the draft with a higher-scoring rewrite, or reports that it kept your version — and never leaves the panel showing a verdict that describes different text.
- [ ] `GET /x/playbook` returns `judgeEffectiveness` with real bucket counts over the whole measured history and no backfill step, gated n≥20, with the `unjudged` bucket separate — and the Playbook tab states the verdict honestly in either direction.
- [ ] `bun scripts/smoke-judge.ts` passes $0 and leaves no rows behind; `--live` adds exactly one real judge call (~$0.003) that persists a 13-dimension row.
- [ ] No surface anywhere sorts, blocks, disables, or refuses on the score.

## Risks / watch items

- **A model grading a model.** If the judge runs on the same provider that drafted the post it will systematically like its own output. Mitigated but not solved: `provider`/`model` are stamped on every row (decision 11), so the bias is measurable later; not forced, because forcing OpenRouter would 503 for an xAI-only setup. Watch the first month of verdicts for suspiciously high `voiceMatch` on drafter-written posts.
- **The rubric's own bait gradient.** `answerEffort` and `strangerAnswerability` reward one-word-answerable posts; read as targets they point at the same `cta_farm` gradient the static-engine study flagged. `negativeRisk` balances them only while a human reads both — which is exactly why decision 4 forbids any surface that reduces the thirteen numbers to one ranking.
- **`judgeEffectiveness` will say "insufficient data" for months.** Two n≥20 bands means ~40 judged-and-measured originals; at ~1 original/day that is a quarter, and only if the judge is actually used. That is the same wait every Playbook cell has had — and the honest alternative to shipping an unvalidated number as advice.
- **Text-hash brittleness is deliberate but will surprise.** Fix a typo after judging and the post reads as `unjudged` forever. That is correct (the verdict described different text) but it means the cell undercounts; the Composer's Apply-all path preserves the link because it re-judges. Watch the `unjudged` bucket's size — if it dwarfs the bands, people are editing after judging and the workflow, not the join, needs changing.
- **Prompt size.** 13 dimensions + penalties + the grounding block is a bigger prefix than any existing stratus prompt except the post drafter. `promptCacheKey` covers repeat calls, but the first call of each niche edit pays full price; keep the instruction block tight and the grounding at the tail.
- **`maxOutputTokens` sizing (JD.4).** 13 integers + a headline + 5 + 5 strings + 12 annotations is a real ceiling; too low and `parseVerdict` correctly returns null but the user has paid for nothing. Size it against a live call during JD.4 and write the number down in a comment.
