## Why / what changes for the user

Scheduling gets a second brain: the audience Active-times heatmap from X Analytics (captured $0 whenever you visit that page) fills in the hours your own measured data can't speak about yet, so "Best time" stops defaulting to earliest-first on thin history. The calendar becomes a week board — seven days, filled slots, open anchors shaded by how good that hour actually is, and a drafts tray that schedules a draft into the best open slot in two clicks. A new **manual** publish mode covers everything the API can't or shouldn't post: posts with Studio visuals, link posts (skipping the $0.20 surcharge entirely) — Chrome notifies you at the slot, Today shows a copy-ready card, and the daily pass links the pasted tweet back to its calendar row so metrics tracking works exactly like API posts. The People dossier and Do-next `dm_ready` items gain "Draft DM": one Grok call grounded strictly on the real shared history (icebreaker discipline), edit, copy, mark sent — logged into the timeline. And `GET /writer` opens a full-width article writing room (explorer-style page, no build step): markdown editor with autosave, outline/section/polish Grok assists grounded on your pillars, winners and playbook guidance, and a "Copy for X" button that puts rich text on the clipboard ready to paste into X's Premium article composer. Throughout, the language contract is **think in any language, publish in English**: every free-text steer in this plan (DM idea/purpose, article idea/instruction, even a body sketched in Romanian) may arrive in any language, and every generated output is English — the reply prompt's "Romanian seed in, English reply out" rule promoted from a one-off to the default for all authoring surfaces.

## Design

**Data (3 new tables + 1 new status, migrations `0013`+):**

```sql
audience_activity (
  id integer PK autoincrement,
  captured_at timestamp_ms NOT NULL,
  metric text NOT NULL,             -- the analytics dropdown at capture time ('likes' …)
  tz_offset_min integer NOT NULL,   -- viewer local tz at capture (grid is local wall-clock)
  cols integer NOT NULL, rows integer NOT NULL,
  grid text NOT NULL                -- JSON number[cols][rows], 0..1 intensity; cols=Mo..Su
)                                    -- append-only: repeated captures = longitudinal series

dm_drafts (
  id text PK uuid, handle text NOT NULL,           -- lowercased, people.handle space
  text text NOT NULL, purpose text,                 -- optional steer that was used
  status text NOT NULL,                             -- draft | sent | discarded
  grounding text,                                   -- JSON: exactly what the model saw (§7.16)
  cost_usd real, created_at, updated_at, sent_at
)

articles (
  id text PK uuid, title text NOT NULL, subtitle text,
  body_md text NOT NULL DEFAULT '', pillar text,    -- validated against active slugs
  status text NOT NULL,                             -- draft | published | discarded
  outline text,                                     -- JSON, persisted by the outline assist
  published_url text, published_at, created_at, updated_at
)
```

`scheduled_posts` gains **no column** — a new status value `manual` (status is free text). Lifecycle: `draft → manual` (scheduledFor required, like pending) `→ posted` via mark-posted or reconcile. The publisher claims only `eq(status,'pending')` (publisher.ts:133) so manual rows are unclaimable **by construction** — the worker file is not touched.

**Pure logic:**
- `extension/src/shared/activeTimes.ts` — `ActiveTimesGrid` type, `parseHeatCells(cellColors, cols, rows)` (color strings → normalized 0..1 grid, min-max scaled), `audienceScoreFor(grid, jsWeekday, hour)` (X columns are Mo..Su; JS `getDay()` 0=Sun — the converter lives here), `gridsEqual` (re-send suppression). Fixture-tested.
- `src/x/posts/manualReconcile.ts` — `matchManualRows(manualRows, publishedRows)`: collapsed-whitespace text equality, window `scheduledFor −1h .. +7d`, `isReply=false`, unlinked rows only (the harvest replies-reconcile matching discipline, §CLAUDE 6.3).
- `extension/src/shared/launch.ts` — `computeManualAlarms(posts, nowMs)`: fire AT `scheduledFor` (no grace — nothing to verify), recently-missed clamps to now, >30min past drops (Today's overdue chip covers it).
- `src/x/articles/prompt.ts` — `ARTICLE_PROMPT_TEMPLATE` (byte-synced with new `article prompt.md`) + `buildArticleAssistInput(mode, tail)` + per-mode structured-output schemas.
- `src/x/people/dm.ts` — `buildDmPrompt(grounding, idea)` + `DM_SCHEMA` (TS-only prompt, icebreakers precedent — no .md).

**Language contract (applies to every Grok prompt in this plan):** all variable-tail inputs (idea, purpose, instruction, selection, notes-verbatim, article body) may be in ANY language — the user often thinks in Romanian; the stable instruction prefix of each prompt states it explicitly ("the steer/idea/body below may be in any language; your output is ALWAYS natural English") — the exact clause the reply prompt already carries for `{{IDEA}}`. Polish/full-draft modes additionally translate non-English source material into English rather than preserving its language. No language detection code anywhere — the rule lives in the prompts.

**Routes (all $0 except the two Grok endpoints):**

```
POST /x/analytics/active-times   {metric, tzOffsetMin, cols, rows, grid} → 201    (validated shape, 0..1 clamp)
GET  /x/analytics/active-times   → latest capture | {capture: null}

POST /x/posts/scheduled/:id/mark-posted  → 200 row   (409 unless status='manual'; flips to posted; NEVER touches posts_published)

POST /x/dms/draft    {handle, idea?, purpose?} → 201 {id, text, grounding}
                     404 unknown_person → 422 no_shared_context → 503 xai_unconfigured → Grok (~$0.005)
GET  /x/dms?handle=&status=      → rows
PATCH /x/dms/:id     {text? | status: sent|discarded}  — sent stamps sentAt + logs manual_dm_logged (existing event type, deterministic id manual_dm_logged:dm_drafts:<id>)

GET/POST /x/articles ; GET/PATCH/DELETE /x/articles/:id      (list excludes body_md; PATCH is the autosave path)
POST /x/articles/:id/assist   {mode: outline|section|polish|full, idea?, heading?, selection?} → proposal JSON
                     503 before spend; outline persists to the row, other modes return text the editor inserts (~$0.01–0.03)

GET /writer          — public-by-design shell (explorer pattern), mounted at root outside the /x guard
```

**Workers:** `dailyMetrics` gains one $0 step after `discover`: `reconcileManualPosts()` — link matched rows (stamp `postedTweetId` + `scheduledPostId` on posts_published side? **no** — stamp `scheduled_posts.postedTweetId` and `posts_published.scheduledPostId`, flip lingering `manual` to `posted`). Publisher untouched.

**Extension:**
- content.ts: active-times capture section gated on `location.pathname.startsWith('/i/account_analytics')`, once per visit, re-send only when grid changed or >12h (`gridsEqual` + local storage stamp), through ApiRequest.
- Composer: publish-mode toggle (API / manual — manual shows "$0 · you paste it" and skips the URL warning), audience-informed "Best time" (blended ranking) + staleness note when capture missing/>28d.
- Calendar: week board rewrite (7-day columns, slot heat = own gated score first / audience shade second, drafts tray with "→ best slot").
- Today: `ManualPostCard` (top area, launch-room styling) driven by session key `manual:due` (background single writer); manual chips in TodayPlan.
- background.ts: manual alarms alongside launch alarms (`stratus-manual:` prefix), notification "Time to post: «…»", messages `stratus/manual-sync` / `stratus/manual-dismiss`.
- People dossier + DoNext: "Draft DM" button → draft view, edit, Copy, "Mark sent".
- `public/writer.html`: self-contained vanilla JS (explorer.html discipline — same localStorage token key), article list rail, markdown editor + preview, autosave PATCH (800ms debounce), assist buttons, word count, "Copy for X" (`ClipboardItem` with text/html + text/plain from a minimal MD→HTML renderer).

**Measurement:** manual posts flow into the existing metrics pipeline automatically (discovery + snapshot — that's the point of the reconcile link, and `has_media` stamps from the attachments field, feeding S0.2's media cell). DM/article volume will be tiny — no playbook cells (below any honest gate); verification is smoke + real-browser checks. Audience-vs-own slot quality is deliberately NOT a new stat: S0.4 best-times already measures outcomes per cell and stays the ranking authority.

## Decisions taken

1. **Article surface = full standalone writer page** (user choice 2026-07-17). Explorer-pattern page at `GET /writer`, articles in a real table, section-level Grok assists. Not a side-panel tab, not generate-only.
2. **Manual-post reminders = Chrome notification + chips** (user choice). C7 alarm machinery extended, not duplicated.
3. **Active-times capture = passive** (user choice). Content script captures when the user happens to visit X Analytics; no tab automation, no timers. Staleness surfaces in the Composer.
4. **`manual` is a status, not a `publish_mode` column.** The publisher claims `status='pending'` only; a status keeps manual rows unclaimable by construction with zero publisher edits. A mode column would put double-post safety one forgotten predicate away.
5. **URL guard does not apply to `manual`** — no API call, no surcharge. This is deliberate product surface: manual mode is the sanctioned way to post links at $0.
6. **mark-posted never writes `posts_published`.** Max stored tweet_id is discovery's since_id checkpoint; inserting a fresh id would skip everything posted between. Linking is the daily reconcile's job (text match), same family as the harvest replies reconcile. A user-edited-before-paste text simply stays unlinked — accepted.
7. **Manual mode v1 is single posts only.** `POST /posts/threads` rejects `manual` (400) — a manual thread is segment-by-segment paste choreography; revisit only if real demand shows.
8. **DM drafting reuses the icebreaker grounding verbatim** (loader extracted, not duplicated) including its refusal ladder — Grok is never allowed to improvise familiarity, and a thin dossier can never spend. DM prompt is TS-only (icebreakers precedent); the .md byte-sync discipline is reserved for the big authoring prompts.
9. **DM drafts persist** (`dm_drafts` + grounding snapshot) — repo habit: continuity beats ephemera; "what did I last send them" is dossier material. Sending stays manual in X; "Mark sent" logs the existing `manual_dm_logged` event through the existing store path.
10. **Audience data ranks strictly below own measured data.** Own gated cells (n≥3) first, audience intensity second, earliest third. The heatmap is who's *online*, not what *worked* — it never overrides a measured cell and is always labeled "audience".
11. **Articles get the byte-sync prompt treatment** (`article prompt.md` ↔ TS literal): they're originals — persona is the product, same reasoning as the post prompt keeping the full §1 biography.
12. **One plan, PLAN.md-flavored** (Phase 10 — Authoring 3.0), with a one-line cross-reference in CIRCLES-PLAN.md for the DM piece.
13. **Multilanguage input, English output — by default, everywhere** (user directive 2026-07-17). Every AI surface this plan adds (DM draft, all four article assist modes) accepts steers and source text in any language and always produces English. This is a prompt-level contract in the stable prefix (byte-sync-tested where the prompt is byte-synced), never runtime language detection; UI placeholders say "any language" so the affordance is discoverable. It generalizes the existing `{{IDEA}}` rule ("Romanian seed in, English reply out") — do not re-litigate toward per-call language options or localized output.

## Done when

1. **Active times:** visiting the X Analytics Audience page lands a row in `audience_activity` ($0, verified in the explorer); the Composer's "Best time" on a thin-history weekday picks the audience-peak anchor instead of earliest-first, and shows the "audience" label + staleness note appropriately.
2. **Manual publish, end-to-end:** schedule a manual post containing a URL (accepted, cost preview $0) → Chrome notification fires at the slot → Today's card copies the text → paste + post in X → "Mark posted" flips the row → next daily pass links `postedTweetId` and the tweet appears in `/x/metrics/posts` like any API post. The publisher provably never claims a due `manual` row.
3. **DM:** from a `responded`-stage person's dossier, "Draft DM" with a **Romanian idea steer** returns an **English** draft grounded only on real shared context (grounding block visible), a stranger with an empty file gets 422 before any spend, and "Mark sent" logs `manual_dm_logged` into the timeline.
4. **Writer:** at `/writer` — create an article, outline assist from a **Romanian idea** fills English sections, a section assist drafts English prose in-place (even over Romanian beat notes), autosave survives reload, and "Copy for X" pastes formatted rich text into X's article composer (real-browser check).
5. **Week board:** the Calendar tab shows 7 days with filled slots, heat-shaded open anchors, and manual/visual chips; a draft schedules into a suggested jittered slot in two clicks.
6. `bun test` + `bun run typecheck` + `bun run lint` green; `scripts/smoke-authoring3.ts` passes at $0.

## Task 1: Active-times parser + shared module (extension, pure)  [parallel-ok]
**Depends on:** none
**Session budget:** ~280 lines, 2 files

**Read first:** codemap header + §5, §7.26–27; `extension/src/shared/earlyReplies.ts` + `.test.ts` (the happy-dom fixture-tested DOM parser exemplar); `extension/src/shared/metricsAria.ts` (locale-hardened parsing + loud-failure conventions); the feature request's Analytics screenshot (7 columns Mo..Su, ~48 half-hour rows, 12am/4am/8am/12pm/4pm/8pm rails, intensity = cell shade).

**Edit:**
- `extension/src/shared/activeTimes.ts` (new) — types + parser + score helpers.
- `extension/src/shared/activeTimes.test.ts` (new).

**How:** Export `interface ActiveTimesGrid { cols: number; rows: number; grid: number[][]; tzOffsetMin: number; metric: string }` (grid indexed `[col][row]`, col 0 = Monday, values 0..1). Two layers, DOM-thin over pure:
- Pure: `parseHeatColors(colors: (string | null)[][], cols, rows): number[][] | null` — parse each CSS color (`rgb()`/`rgba()`/hex) to per-cell luminance-or-alpha scalar, then **min-max normalize across the grid** (the absolute palette is X-owned and unknowable; relative intensity is what we need). All-equal grid → all 0.5. Any unparseable color → that cell 0 + count; >20% unparseable → return null (fail closed, `metrics_unparsed` spirit). Also `audienceScoreFor(grid: ActiveTimesGrid, jsWeekday: number, hour: number): number | null` — map JS weekday (0=Sun) to column (`(jsWeekday + 6) % 7`), average the row(s) covering that hour (`rows === 48` → rows `hour*2, hour*2+1`; `rows === 24` → row `hour`; other row counts → proportional), null when out of range. And `gridsEqual(a, b, epsilon = 0.02)`.
- DOM-thin: `extractActiveTimesSection(root: Document | Element): { colors: (string|null)[][], cols, rows } | null` — locate the section whose heading text matches `/active times/i`, collect the heat cells **by structure** (the grid of uniform small divs between the weekday footer labels and the hour rail), reading `style.backgroundColor` inline style first, `getComputedStyle` fallback. Keep every selector in named constants at the top — this WILL need live adjustment (risk item). Return null when the shape isn't a plausible grid (cols ≠ 7, rows < 12).
Module stays dependency-free (inlined into the content IIFE).

**Tests:** pure layer thoroughly — normalization (min-max, all-equal → 0.5, unparseable-cell counting, >20% → null); rgb/rgba/hex parsing; `audienceScoreFor` weekday mapping (Mon col 0 ↔ jsWeekday 1; Sun col 6 ↔ 0), 48-row hour averaging, 24-row passthrough, out-of-range null; `gridsEqual` epsilon. DOM layer: happy-dom fixture with inline-styled cells → full round-trip to a normalized grid; a fixture with a broken section → null.

**Done when:**
- [ ] Pure parse/score/equality helpers fully covered; DOM extraction round-trips a fixture
- [ ] `bun test` + `bun run typecheck` + `bun run lint` green
- [ ] Committed: `feat(extension): active-times heatmap parser (shared, fixture-tested)`

**Cost note:** $0 — pure parsing.
