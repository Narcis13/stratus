# Niche — first-class identity + strategy container (N0)

- **Status:** planned 2026-07-16 · not started
- **Goal fit:** serves goals 1, 3, 4 — the drafters (goal 1), voice/swipe strategy (goal 3), and the people/doctrine layer (goal 4) all currently hardcode ONE person's niche. This phase abstracts that configuration into a managed entity, the same move §8.6 made for pillars. The "future multi-user product" is motivation, not scope: nothing multi-tenant is built.
- **Cost impact:** $0 recurring. Niche wizard ~$0.01 per click (one Grok structured-outputs call, runtime-gated). No X API reads or writes anywhere in this plan. One-time prompt-cache misses after cache-key changes (~cents).
- **Invariants touched:** §7.13 (sync SQLite — activation swap is a sync txn); §7.14 (byte-sync prompts — both .md files and TS literals change together, tests adapt); §7.15/16 (persona is constant-per-niche so it substitutes IN PLACE, not at the variable tail — cache prefix stays stable; niche fields are server-stamped, never client-parsed); §7.19 (no new stats; doctrine knobs feed existing gated stats unchanged); §7.20 (new routes are static paths only — no `:param` collision risk, but `/x/niches/:slug` must mount so it can't shadow anything); §7.22 (wizard = always-mounted route + runtime 503 without XAI key); §7.29/30 (docs sync + smoke script).
- **Codemap sections relevant:** §3.3 (posts/prompt.ts, posts/pillars.ts, posts/pillarDraft.ts, replies/prompt.ts), §3.4 (pillars.ts, channels.ts, drafter.ts, replies.ts, brief.ts, voice.ts, followups.ts, playbook.ts, digest.ts), §4 (contentPillars L33, channels L54), §5 (Settings.tsx, api.ts), §7, §10.

## Why / what changes for the user

Today "the niche" is scattered and hardcoded: the 51-year-old-builder biography lives inside two byte-synced prompt templates, the beliefs block inside one of them, the doctrine numbers (10–20 replies/day, 70/30 ratio, 2–10x target band) as constants in three routes, and the prose self-description in a dead file (`src/my_niche.md`). After this plan, **Niche is a first-class DB entity**: the Settings tab shows the active niche — persona, beliefs, reply persona, description, doctrine knobs — all editable without a deploy, exactly like pillars became editable in §8.6. Pillars and channels are owned by a niche. A second niche (e.g. nutrition) can be created, given its own persona/pillars/channels, and activated — and no builder biography, builder pillar, or builder channel leaks into its drafts. An AI wizard turns a pasted prose description into a complete proposed niche (persona + beliefs + pillars + channels) for review-then-save.

## Design

**Hierarchy decided:** niche → owns pillars (output taxonomy) + channels (input taxonomy) + persona/beliefs (prompt grounding) + doctrine (the REPLY-GUIDE numbers). Channels stay shallow topic rooms; niche sits above them. Exactly ONE active niche at a time (v1); data tables (voice, people, drafts, calendar) are deliberately NOT partitioned yet — but reply `contextSnapshot` gets a niche stamp from day one so future per-niche analytics are free.

**Data** — new table + two ownership columns (one migration, `0013`):

```ts
export const niches = sqliteTable('niches', {
  slug: text('slug').primaryKey(),                 // PILLAR_SLUG_RE rules reused
  label: text('label').notNull(),
  description: text('description'),                // prose self-description (my_niche.md style)
  persona: text('persona').notNull(),              // → {{PERSONA}} (post prompt §1 body)
  beliefs: text('beliefs').notNull(),              // → {{BELIEFS}} (post prompt §5 body)
  replyPersona: text('reply_persona').notNull(),   // → {{REPLY_PERSONA}} (reply prompt "Who I am" body)
  doctrine: text('doctrine', { mode: 'json' }).$type<Partial<NicheDoctrine>>(), // null = all defaults
  active: integer('active', { mode: 'boolean' }).default(false).notNull(),
  createdAt / updatedAt,                           // contentPillars pattern
});
// + contentPillars.niche: text('niche')  — nullable; migration backfills 'builder'
// + channels.niche: text('niche')        — nullable; migration backfills 'builder'
```

Seed (migration `INSERT OR IGNORE`, mirrored by `DEFAULT_NICHE` in code — the §8.6 `DEFAULT_PILLARS` discipline): slug `builder`, active `true`, persona/beliefs/replyPersona lifted **verbatim** from the current prompt sections, description lifted from `src/my_niche.md`.

```ts
interface NicheDoctrine {
  replyTargetMin: number;    // 10  (brief.ts REPLY_TARGET.min)
  replyTargetMax: number;    // 20  (brief.ts REPLY_TARGET.max)
  weekReplyTargetPct: number;// 70  (brief.ts WEEK_REPLY_TARGET_PCT)
  targetBandMinX: number;    // 2   (voice.ts targetBand)
  targetBandMaxX: number;    // 10
}
```

**Pure logic** — `src/x/niche/` (pillars.ts pattern: lowest layer, no route imports):
- `defaults.ts`: `DEFAULT_NICHE`, `DEFAULT_DOCTRINE`, `NicheDoctrine`, `resolveDoctrine(row)` (field-by-field lenient merge over defaults — brandKit parse spirit).
- `store.ts`: `loadActiveNiche()` (active row, else `DEFAULT_NICHE` when the table is empty — fresh-DB/pre-migration fallback like `getActivePillars`), `loadActiveNicheSafe()` (never throws — a niche-layer failure yields a default-grounded draft, never a failed one; §7.8 spirit), `loadDoctrine()`.

**Prompt architecture** (the deep part): post prompt.md §1 and §5 keep their headings but their bodies become `{{PERSONA}}` / `{{BELIEFS}}` placeholders (§4-pillars precedent: the .md section becomes a pointer). Reply prompt.md "## Who I am" body becomes `{{REPLY_PERSONA}}`. Substitution happens **in place, not at the variable tail** — deliberate deviation from §7.15, justified: persona is constant per niche, so the substituted prefix is byte-stable across calls and the cache prefix survives. Cache keys gain a niche suffix (`` `${PROMPT_CACHE_KEY}:${niche.slug}:${niche.updatedAt.getTime()}` ``) so editing the niche cleanly busts the cache. `buildBatchGrokInput`'s `voiceBlock()` slice (between `## Who I am` and `## The two variants`) now carries the placeholder — the batch builder substitutes the same block, so single and batch can never drift. Byte-sync tests keep asserting .md ↔ TS literal identity (both now contain the tokens). A new **equivalence test** asserts that building with `DEFAULT_NICHE` yields a prompt containing every original biography/beliefs/reply-persona line verbatim — the "nothing changed for the current user" guarantee.

**Routes** — new `src/x/routes/niche.ts`, always mounted in `mountX` (before nothing sensitive; static paths only):

| Method + path | Semantics |
|---|---|
| `GET /x/niche` | Active niche + resolved doctrine. Falls back to `DEFAULT_NICHE`. |
| `GET /x/niches` | All niches, sortOrder-free, active first. |
| `POST /x/niches` | Create (validated slug/lengths; created **inactive**). 409 `slug_exists`. |
| `PATCH /x/niches/:slug` | Edit fields/doctrine. `active:true` → one sync txn: deactivate all, activate this. `active:false` on the only active → 409 `last_active_niche` (pillars 409 pattern). |
| `DELETE /x/niches/:slug` | 409 `niche_active` if active; else delete. Owned pillars/channels keep their stamp (orphan-tolerant, channels-delete pattern). |
| `POST /x/niche/draft` | Wizard: `{description (≤5000)}` → ONE Grok structured-outputs call → proposal `{slug,label,persona,beliefs,replyPersona,pillars[3],channels[≤5 with keywords]}`. **Never persisted** (pillarDraft.ts pattern). Runtime XAI check → 503. ~$0.01. |

**Consumers rewired:** `drafter.ts` + `pillarDraft.ts` + `replies.ts` (single + batch) load the active niche server-side and substitute; `pillarDraft.ts`'s hardcoded `PERSONA` const (L40) is replaced by `niche.persona + description`. `brief.ts` REPLY_TARGET / WEEK_REPLY_TARGET_PCT and `voice.ts` `targetBand` read doctrine (with today's values as defaults — every existing test unchanged). Pillars/channels routes stamp `niche` on create and filter reads to the active niche (`OR niche IS NULL` legacy tolerance). Drafter refuses `409 no_pillars_for_niche` when a non-default active niche has zero active pillars (falling back to `DEFAULT_PILLARS` would leak builder pillars into a nutrition niche — the exact interference this plan exists to prevent).

**Extension:** Settings tab gains a **Niche** card (active-niche editor + niche list + wizard). `api.ts` gains `api.niche.*`. No content-script changes.

**MCP:** one curated tool `x_niche` (in-process `GET /x/niche`) → 17 tools.

**Measurement:** `contextSnapshot.niche = {slug}` stamped on every reply draft (server-stamped, §7.16) — the future per-niche crosstab key. Equivalence test + smoke script are the correctness measures; no new gated stats (nothing statistical shipped here).

## Decisions taken

1. **V1 scope = single active niche** (user-confirmed). No data-row partitioning of voice/people/drafts/calendar. Taxonomy tables (pillars, channels) DO get ownership columns now because they're cheap, and reply snapshots get the niche stamp — both make the future partition a backfill, not a rework.
2. **Full persona extraction** (user-confirmed): post §1 + §5 and the reply 3-fact persona become niche fields. The "How I/the replies sound" voice-rule sections stay in the templates — they're niche-agnostic anti-slop craft, not identity.
3. **Niche above channels** (user-confirmed): new `niches` entity; channels stay shallow. NOT extending the channels table.
4. **In-place substitution, not variable tail**, for persona blocks — constant-per-niche content keeps the cache prefix stable; tail placement would put identity after per-call content for no cache benefit. Cache key carries slug+updatedAt to bust on edit.
5. **Zero-pillar refusal over fallback** for non-default niches: `DEFAULT_PILLARS` fallback survives ONLY when the active niche is `builder` (fresh-DB compat). Otherwise 409 — cross-niche leakage is worse than a refused draft.
6. **Doctrine = 5 knobs only** (reply quota min/max, week reply %, band min/max multipliers). BAND thresholds, playbook gates, quest definitions, cadence ladders stay global — they're method, not niche. Revisit only if a real second niche demands it.
7. **Wizard proposes, human persists** — same contract as `POST /x/pillars/draft`.
