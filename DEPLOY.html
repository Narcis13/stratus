1. Hetzner tooling — what you actually need

There are three serious automation surfaces for Hetzner Cloud. Pick based on what you'll do most:

ToolWhen to useEffort
hcloud CLI One-shot ops from your laptop: list/create/snapshot servers, manage SSH keys + firewalls, read pricing. Best for solo workflow. Low — install + paste API token, done.
Terraform (hetznercloud/hcloud) You want infra defined in code, re-creatable from scratch. Overkill for one server you'll keep for months. Medium — extra DSL to learn.
cloud-init via --user-data-from-file Bootstrap a brand-new server at creation time. Doesn't apply here — your CX23 already exists. One YAML file, no agent on host.

For this project: install hcloud CLI locally (server management), and use the two bash scripts below to provision + deploy. No Terraform, no Ansible.

1a. Install hcloud on your Mac local

brew install hcloud
# create an API token in the Hetzner Cloud Console:
#   Project → Security → API tokens → Generate API token (Read & Write)
hcloud context create stratus     # paste token when prompted
hcloud server list                # should show ubuntu-4gb-nbg1-1

From here you can hcloud server poweroff, hcloud server create-image (snapshot), hcloud firewall create, etc. — without ever opening the web console.

2. Prerequisites

2a. SSH key on the server local

You said you can already log in as root. If you're still using a password, push your laptop's public key now so the scripts below can run unattended:

ssh-copy-id root@116.203.39.245
# then test password-less:
ssh root@116.203.39.245 'echo ok'

2b. Pick a hostname (strongly recommended)

X requires HTTPS for the OAuth callback (except 127.0.0.1), and you don't want to ship bearer tokens over plain HTTP. Two paths:

  • Real domain (e.g. api.yourdomain.com) — point an A record at 116.203.39.245. Caddy will get a Let's Encrypt cert automatically.
  • No domain? Use 116-203-39-245.sslip.io for free — it's a wildcard DNS service that resolves the IP back. Caddy can issue a real cert against it.

Throughout this doc, substitute your choice for $DOMAIN.

2c. Your .env values ready

You'll need these (you already have them locally in .env):

KeySource

X_CLIENT_SECRETconsole.x.com, your app
X_OAUTH_REDIRECT_URIhttps://$DOMAIN/auth/x/callback (server) — see §6

Tokens already in Neon. Phase 1 swapped .tokens.json for a Postgres-backed token store. Because you'll point the server at the same DATABASE_URL, no token copy is needed — the workers read the existing row on first tick.

3. Server bootstrap — one script server

Run this once per server. It installs Bun, Caddy, a hardened firewall, a non-root stratus user, and a systemd unit. Idempotent — safe to re-run.

Save the following as scripts/bootstrap.sh in your repo (file does not exist yet — create it):

#!/usr/bin/env bash
# scripts/bootstrap.sh — run ONCE on a fresh Hetzner Ubuntu box.
#   usage: DOMAIN=api.example.com ./scripts/bootstrap.sh root@116.203.39.245
set -euo pipefail

REMOTE="${1:?usage: bootstrap.sh user@host}"
DOMAIN="${DOMAIN:?set DOMAIN=api.example.com (or 116-203-39-245.sslip.io)}"

ssh "$REMOTE" DOMAIN="$DOMAIN" bash -s <<'REMOTE_SCRIPT'
set -euo pipefail

echo "==> apt update + base packages"
export DEBIAN_FRONTEND=noninteractive
apt-get update -y
apt-get install -y curl unzip git ufw ca-certificates debian-keyring debian-archive-keyring apt-transport-https

echo "==> firewall (ufw)"
ufw --force reset
ufw default deny incoming
ufw default allow outgoing
ufw allow 22/tcp
ufw allow 80/tcp
ufw allow 443/tcp
ufw --force enable

echo "==> create 'stratus' user"
id -u stratus >/dev/null 2>&1 || useradd -m -s /bin/bash stratus
install -d -o stratus -g stratus -m 0750 /home/stratus/app
install -d -o stratus -g stratus -m 0700 /home/stratus/.config

echo "==> install Bun as 'stratus'"
sudo -u stratus -H bash -lc '
  if ! command -v bun >/dev/null; then
    curl -fsSL https://bun.sh/install | bash
  fi
  echo "export PATH=\"\$HOME/.bun/bin:\$PATH\"" > ~/.profile
'
ln -sf /home/stratus/.bun/bin/bun /usr/local/bin/bun

echo "==> install Caddy (official repo)"
if ! command -v caddy >/dev/null; then
  curl -1sLf 'https://dl.cloudsmith.io/public/caddy/stable/gpg.key' \
    | gpg --dearmor -o /usr/share/keyrings/caddy-stable-archive-keyring.gpg
  curl -1sLf 'https://dl.cloudsmith.io/public/caddy/stable/debian.deb.txt' \
    > /etc/apt/sources.list.d/caddy-stable.list
  apt-get update -y
  apt-get install -y caddy
fi

echo "==> write Caddyfile for $DOMAIN"
cat >/etc/caddy/Caddyfile <<EOF
${DOMAIN} {
  encode zstd gzip
  reverse_proxy 127.0.0.1:3000
}
EOF
systemctl reload caddy

echo "==> systemd unit"
cat >/etc/systemd/system/stratus.service <<'UNIT'
[Unit]
Description=stratus API (Bun + Hono)
After=network-online.target
Wants=network-online.target

[Service]
Type=simple
User=stratus
Group=stratus
WorkingDirectory=/home/stratus/app
EnvironmentFile=/home/stratus/app/.env
ExecStart=/usr/local/bin/bun run src/app.ts
Restart=on-failure
RestartSec=3
# crash safety
LimitNOFILE=65535
# hardening
NoNewPrivileges=true
ProtectSystem=strict
ProtectHome=false
ReadWritePaths=/home/stratus/app
PrivateTmp=true

[Install]
WantedBy=multi-user.target
UNIT

systemctl daemon-reload
systemctl enable stratus.service
echo "==> bootstrap done. Now run deploy.sh."
REMOTE_SCRIPT

Run it from your laptop:

chmod +x scripts/bootstrap.sh
DOMAIN=116-203-39-245.sslip.io ./scripts/bootstrap.sh root@116.203.39.245
# or DOMAIN=api.yourdomain.com ./scripts/bootstrap.sh root@116.203.39.245

SSH from a non-root user later. Once the app deploys, you can disable root password login: sed -i 's/^#*PermitRootLogin.*/PermitRootLogin prohibit-password/' /etc/ssh/sshd_config && systemctl reload ssh. Leave it for now if you're still debugging.

4. App deploy — one script local

This is the script you'll run again and again every time you push changes. It rsyncs the repo, installs deps on the server, and restarts the service.

Save as scripts/deploy.sh:

#!/usr/bin/env bash
# scripts/deploy.sh — push current working tree to the server and restart.
#   usage: ./scripts/deploy.sh root@116.203.39.245
set -euo pipefail

REMOTE="${1:?usage: deploy.sh user@host}"
APPDIR=/home/stratus/app

echo "==> rsync code (excluding node_modules, dist, .env, tokens)"
rsync -az --delete \
  --exclude '.git' \
  --exclude 'node_modules' \
  --exclude 'extension/node_modules' \
  --exclude 'extension/dist' \
  --exclude '.tokens.json' \
  --exclude '.env' \
  --exclude '.env.local' \
  ./ "$REMOTE":"$APPDIR/"

echo "==> fix ownership"
ssh "$REMOTE" "chown -R stratus:stratus $APPDIR"

echo "==> upload .env if missing (one-time)"
ssh "$REMOTE" "test -f $APPDIR/.env" || scp .env "$REMOTE":"$APPDIR/.env"
ssh "$REMOTE" "chown stratus:stratus $APPDIR/.env && chmod 600 $APPDIR/.env"

echo "==> bun install + restart"
ssh "$REMOTE" "sudo -u stratus -H bash -lc 'cd $APPDIR && bun install --frozen-lockfile'"
ssh "$REMOTE" "systemctl restart stratus.service"

echo "==> health check"
sleep 2
ssh "$REMOTE" "systemctl is-active stratus.service"
ssh "$REMOTE" "curl -fsS http://127.0.0.1:3000/healthz && echo"
echo "==> deployed."

Run:

chmod +x scripts/deploy.sh
./scripts/deploy.sh root@116.203.39.245

Edit .env on the server before first start to set X_OAUTH_REDIRECT_URI=https://$DOMAIN/auth/x/callback. Everything else (Neon URL, X creds, API_TOKEN) can stay identical to local.

4a. Smoke-test from your laptop

curl -s https://$DOMAIN/healthz                              # expect: ok
curl -s -H "Authorization: Bearer $API_TOKEN" https://$DOMAIN/x/calendar
# expect: a JSON array (possibly empty)

5. Wire up the Chrome extension local

The side panel reads its API base URL from the Settings tab — no rebuild needed, just:

  1. Open the extension side panel → Settings.
  2. Set API base URL to https://$DOMAIN (no trailing slash).
  3. Set Bearer token to your API_TOKEN from .env.
  4. Click Save, then reload the side panel.

The calendar/replies/drafts tabs will start pulling from the Hetzner box.

CORS. The corsMiddleware already allows any chrome-extension://*. If you ever build a separate web UI, add its origin to ALLOWED_ORIGINS in .env (comma-separated) and restart.

6. X OAuth callback (only if re-authing on the server)

Your existing refresh token in Neon will continue to rotate on each call — you generally don't need to re-auth. But if you ever do (token revoked, account swap, etc.), update the callback URL in the X dev portal first:

  1. console.x.com → your app (Production env) → User authentication settings.
  2. Callback URI list — add https://$DOMAIN/auth/x/callback.
  3. Set X_OAUTH_REDIRECT_URI in the server's .env to match exactly.
  4. SSH in, run the OAuth flow once: sudo -u stratus -H bash -lc 'cd /home/stratus/app && bun run auth' — but bind it to a local port and reverse-proxy temporarily, or just keep auth on your laptop with 127.0.0.1 and let the resulting refresh token sync via Neon. The latter is simpler.

Backups + snapshots (via hcloud) local

# one-off snapshot before risky work
hcloud server create-image ubuntu-4gb-nbg1-1 --type snapshot --description "pre-phase-5"

# enable Hetzner's automatic daily backups (€0.80/mo extra on CX23)
hcloud server enable-backup ubuntu-4gb-nbg1-1

# list / delete snapshots
hcloud image list --type snapshot
hcloud image delete <id>

Neon already snapshots Postgres on its side, so backups here are just for system config + the codebase.

Firewall (via hcloud, replaces UFW if you prefer cloud-side)

hcloud firewall create --name stratus-web
hcloud firewall add-rule stratus-web --direction in --protocol tcp --port 22 --source-ips 0.0.0.0/0 --source-ips ::/0
hcloud firewall add-rule stratus-web --direction in --protocol tcp --port 80 --source-ips 0.0.0.0/0 --source-ips ::/0
hcloud firewall add-rule stratus-web --direction in --protocol tcp --port 443 --source-ips 0.0.0.0/0 --source-ips ::/0
hcloud firewall apply-to-resource stratus-web --type server --server ubuntu-4gb-nbg1-1

The Hetzner-side firewall runs before traffic ever hits the VM — UFW becomes redundant once this is in place. Keep both initially; tighten later.
